Trengo ¬∑ WhatsApp Business API

Real-Time Owner & Guest Communication at Scale

üî¥ The Problem

Nice2stay property owners were receiving booking confirmations, inquiry alerts, and availability updates via email ‚Äî leading to response delays of 2‚Äì6 hours. Guests expected instant confirmations, and slow communication was directly causing booking drop-offs and owner dissatisfaction at scale.

üîß How We Built It

We integrated the Trengo WhatsApp Business API into the Rails backend via webhooks and a dedicated message-dispatch service. Sidekiq workers were configured to trigger WhatsApp notifications on booking events (confirmed, cancelled, modified). We built template management in the admin panel and used Trengo's unified inbox to centralize all owner-guest conversations. Dynamic variables (guest name, property, dates, amount) were injected into approved WhatsApp templates automatically on every booking event.

‚úÖ Result in Production

Owner response time dropped from hours to under 60 seconds. Guest booking confirmation became instant. The centralized Trengo inbox eliminated fragmented communication across email threads, improving satisfaction on both sides of every booking.

OpenGDS ¬∑ Channel Manager

Real-Time Property Availability Sync Across All Channels

üî¥ The Problem

Nice2stay properties were listed on multiple booking channels (Airbnb, Booking.com, direct site). Without a unified availability layer, double bookings and stale inventory data were a recurring problem ‚Äî leading to manual reconciliation, owner complaints, and direct revenue loss across the platform.

üîß How We Built It

We integrated OpenGDS as the central channel distribution layer. The Rails backend pushed availability and rate updates to OpenGDS via their REST API, which propagated changes across all connected OTAs in real time. We built a Sidekiq pipeline to handle 120K+ daily sync jobs ‚Äî batching property calendars, rate plans, and minimum-stay rules. Error handling with retry logic ensured no sync was silently dropped, and a sync dashboard gave the ops team full visibility into job status per property.

‚úÖ Result in Production

Double bookings dropped to near-zero. Availability accuracy across all channels reached real-time parity. The ops team reclaimed hours previously spent on manual calendar reconciliation, and owner confidence in the platform increased significantly.

ElasticSearch v6 ‚Üí v8

Migrating 100K+ Listings Search with Zero Downtime

üî¥ The Problem

Nice2stay ran ElasticSearch v6, which had reached end-of-life. Search performance was degrading as the index grew past 100K+ property listings. Complex geo-filtering, multi-language queries, and availability-aware search were slow and returning irrelevant results ‚Äî directly impacting user conversion rates and SEO performance.

üîß How We Built It

We ran a parallel migration strategy: stood up a new ES v8 cluster alongside the existing v6, re-indexed all 100K+ listings with updated mappings (geo-point fields, multi-language analyzers, nested availability ranges), and used Searchkick on the Rails side to abstract the query layer. We shadow-wrote to both clusters during transition, validated result parity across hundreds of test queries, then cut over with a single config switch ‚Äî zero downtime, zero data loss.

‚úÖ Result in Production

Search response times improved by over 40%. Geo-filter accuracy and relevance scoring were significantly enhanced, delivering more relevant property results to users. The entire migration completed with zero downtime and zero data loss across all 100K+ listings.

System Architecture ¬∑ N2S Platform Rebuild

Dual-Repo Monolith ‚Üí Centralised API-Driven Architecture

üî¥ The Problem

Nice2stay originally ran as two completely separate monolithic repositories: the guest-facing booking site and the internal CRM ‚Äî each with its own database. A background sync service was responsible for keeping both databases consistent, but under heavy load the sync would lag, drop jobs silently, and produce data inconsistencies between the two sides. The result was manual database patching by engineers, both servers running at full capacity constantly, and zero ability to scale either system independently. As property inventory crossed 100K listings, this architecture had become a critical bottleneck.

üîß How We Built It

We designed and executed a full architectural migration. First, we centralised both databases into a single PostgreSQL source of truth, eliminating the sync service entirely and with it all data inconsistency risk. We then exposed all business logic through a unified Rails 7 API layer that both frontends consumed. The guest-facing booking site was rebuilt in Nuxt.js 3 with full server-side rendering for SEO performance and Core Web Vitals improvement. The internal CRM was rebuilt as a standalone Vue 3 SPA consuming the same API ‚Äî giving each frontend its own deployment pipeline and release cadence without touching shared infrastructure. Both repos were migrated incrementally under a feature-flag strategy to maintain zero-downtime throughout.

‚úÖ Result in Production

Data inconsistencies dropped to zero ‚Äî no more manual patching. Server load was reduced significantly as the redundant sync jobs were eliminated. The Nuxt.js SSR migration delivered a 40% improvement in page load speed and a 35% increase in organic SEO traffic. Each team could now deploy the CRM and the guest site independently, cutting release cycle time and unblocking parallel development across both products.
